#### 2. Provenance and structure are strong

#### 3. Dogfooding is producing the right kind of pressure

### What is bad

#### 1. The operator experience is too fragmented
Too many actions that belong to one human intention are split across too many commands.

This is the single most obvious usability problem.

#### 2. Discovery is too failure-driven
The system teaches the operator by rejecting them.

That is bad CLI design.
It is worse agent UX.

The user should not need to guess the valid intent, campaign, principal, or required fields by failing forward.

#### 3. The two main surfaces are not yet proper peers
The TUI and CLI feel like two incomplete halves, not one coherent organism.

The TUI can do things the CLI cannot see cleanly.
The CLI can do things the TUI cannot initiate cleanly.
That breaks stigmergy.

#### 4. The system currently over-serializes governance
The graph model may be rich for good reasons.
The operator path does not need to expose every layer as a separate chore.

Right now, it does.

#### 5. Some architecture is drifting against declared invariants
The snapshot/projection layer looks suspiciously like bedrock duplication.
That is not a small concern.
That is exactly how systems start violating their own doctrine.

## How XYPH could be more stigmergic

The word is **stigmergic**, and yes, that is exactly the right frame.

A stigmergic system works when one actor leaves traces in the environment and another actor can discover and act on them naturally, without needing out-of-band coordination.

Right now XYPH is only partially there.

### What full stigmergy would look like

- a human asks for help in the TUI and the agent sees that naturally in the CLI
- an agent files a suggestion and the human sees it in the TUI without special filtering
- “what changed?” is always answerable from the shared graph state
- the system has one obvious queue for attention, not five partial ones
- surfaces differ in presentation, not in what reality they can perceive

### Current failure modes

- ask-ai work exists in the graph but is not naturally visible to the CLI flow
- suggestions split across different internal arrays cause surface blindness
- discovering valid references often requires separate exploratory commands or scripts
- neither surface is fully self-sufficient

### How to improve stigmergy immediately

#### 1. Build one unified attention queue
This should combine:
- ask-ai jobs
- suggestions
- diagnostics
- stale in-progress work
- governance blockers
- human-authored requests
- agent-authored requests

One place to look.
One place to act.

#### 2. Enforce surface parity
If TUI writes something, CLI must see it.
If CLI writes something, TUI must see it.
If the graph contains it, both surfaces must expose it in some intelligible way.

#### 3. Make cross-surface writes first-class
The system should know whether an artifact was created by a human through TUI, a human through CLI, an agent through CLI, or another API surface.
Not to separate them, but to preserve context while keeping shared visibility.

#### 4. Prefer queues over hidden categories
Every time the operator has to remember which lane/view/array/type contains a thing, XYPH becomes less stigmergic.

The environment should advertise the trace.
That is the point.

## What I would change today

If I had authority to redirect effort immediately, I would prioritize these changes:

### 1. Build a composite cycle-start flow
Not because the graph should become simpler.
Because the operator path must.

Starting a small cycle should not require a ritual of independent commands for every intermediate object if XYPH can safely scaffold them.

Keep the graph rich.
Compress the path.

### 2. Build one real query surface
A serious `search` / `find` / `next` / `show` family should exist and it should make scripting around XYPH unnecessary for common read tasks.

If I need Python to ask “what exists?” or “what is next?”, the product is not done.

### 3. Fix TUI↔CLI visibility as a hard requirement
This is not polish.
This is core.

Without this, the system is not truly graph-coordinated across surfaces.

### 4. Audit the snapshot/projection layer aggressively
Every part of it should justify itself against bedrock-boundary.

If it is product meaning, keep it.
If it is bedrock duplication, kill it.

### 5. Make merge honor real CI truth
A sealed thing that bypassed actual CI is suspicious.
That should not remain ambiguous.

### 6. Add project profiles
Hours vs t-shirt sizes vs none.
Strict governance vs compressed solo flow.
Checkpoint push policy.
Required traceability depth.

These should be project-configurable, not hardcoded doctrine.

## COOL IDEAS™

### 1. Solo mode that preserves graph truth but compresses operator ceremony
Not “turn governance off.”
Not “fake a shortcut.”
A real profile that:
- scaffolds the full traceability chain
- preserves provenance
- preserves the graph model
- removes command-level drudgery

### 2. Intent-router CLI
Instead of forcing the operator to remember every low-level verb, let XYPH support intent-first actions:

- start this task
- close this cycle
- create backlog item
- show what needs attention
- explain why this action is blocked

Not open-ended chatbot mush.
Structured intent routing.

### 3. Explain-blockers as first-class product surface
Every blocked action should produce:
- the exact reason
- missing artifacts
- valid next commands
- optionally a scaffold command that fixes the missing chain

This would be a giant quality-of-life upgrade.

### 4. Unified “attention inbox”
A real cross-surface inbox for:
- AI requests
- human suggestions
- system diagnostics
- stale work
- action-needed governance items

Not just another dashboard widget.
A serious coordination primitive.

### 5. Surface parity tests
Tests that literally assert:
- TUI-created items appear in CLI-visible queues
- CLI-created items appear in TUI-visible queues
- suggestion families remain visible through both read models
- graph-visible artifacts cannot silently disappear from one surface

This would kill an entire class of bugs.

### 6. Generated operator scaffolds from schema + policy
Given the graph schema and active project profile, XYPH should be able to scaffold:
- cycle starts
- required traceability chains
- valid reference choices
- review flows
- evidence stubs

This could make the system feel intelligent without pretending to be magical.

### 7. Feedback-native mode
A session like this should be easier to mine automatically.

XYPH should be able to ingest:
- command failures
- repeated help lookups
- retry loops
- script-arounds
- graph-only pushes that trigger full tests

and turn those into structured feedback candidates automatically.
