вқҜ What is the GraphSnapshot? Jesus is this another "XPHY is caching the fucking 
 git-warp?"                                                                     

вҸә Good instinct. Let me lay out exactly what it is.

  GraphSnapshot is a 1700-line read model built by ObservedGraphProjection.ts.
  It:

  1. Queries git-warp for every node type in parallel (task:*, campaign:*,
  intent:*, suggestion:*, etc.)
  2. Resolves edges into denormalized fields (e.g., walks belongs-to edges to
  set questNode.campaignId)
  3. Builds typed arrays (quests[], campaigns[], intents[], aiSuggestions[],
  etc.)
  4. Computes derived data (topological sort, transitive downstream counts,
  completion summaries)
  5. Returns one big object that the CLI and TUI render from

  So yes вҖ” it is a denormalized projection layer sitting between git-warp and 
  the UI. Every time git-warp's graph gets a new entity type, someone has to:
  - Add a query to the parallel batch (line ~443)
  - Add a builder loop to project the raw props into a typed model
  - Add the array to the GraphSnapshot interface
  - Update every consumer (CLI dashboard, TUI cockpit, etc.)

  And when someone forgets a consumer вҖ” like aiSuggestions not being wired into
  status --view suggestions вҖ” you get invisible data.

  The question you're really asking: should this exist at all? git-warp already
  has:
  - graph.query().match('suggestion:*').where({ status: 'queued' 
  }).select(['id', 'props']).run() вҖ” typed, filtered queries
  - graph.neighbors() вҖ” edge traversal
  - graph.traverse.* вҖ” computed graph analysis
  - graph.subscribe() / graph.watch() вҖ” reactive updates

  The snapshot is doing what git-warp already does, just pre-computing it into a
   big denormalized blob. The only value-add is the edge denormalization
  (turning belongs-to edges into campaignId fields) and the derived computations
   (topo sort, blocking counts). But those could be pure functions over git-warp
   query results instead of a monolithic 1700-line projection.
